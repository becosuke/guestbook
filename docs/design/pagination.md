# Pagination

`ListPosts` のページネーション規約は [AIP-158](https://google.aip.dev/158) に従う。

## ソート順

`Post` は以下の順序で列挙される。

1. `create_time` の降順（新しい投稿が先）
2. 同 `create_time` の場合は `post_id` の昇順

このソート順はサーバ固定であり、クライアントから変更する手段は提供しない。

## page_size

- `int32`。**0 以上**を受け付ける。
- protovalidate のルールは `int32.gte = 0`。負数は `INVALID_ARGUMENT` で拒否される。
- `0`（または proto3 の数値ゼロ値として未指定）はバリデーションを通過し、usecase 層で **サーバデフォルト 10 件** に置き換えられる。AIP-158 が認める「サーバが最終件数を決めてよい」枠を活用した挙動。
- 明示的に `1` 以上の値が指定されたときはその値をそのまま使う。

## page_token

- 不透明（**opaque**）な文字列。クライアントは中身を解釈してはならず、
  受け取った値をそのまま次のリクエストに渡す責務のみを持つ。
- 空文字（`""`）は「最初のページ」を意味する。
- 実体は UUID 形式の `pagination_id` 文字列だが、これはサーバの実装詳細であり、
  クライアントから見ると不変ではない契約。

## next_page_token

- 続きのページが存在する場合、次の `page_token` として使うべき不透明な文字列を返す。
- **空文字** の場合は「これ以上ページが無い（終端）」を意味する。

## カーソル方式（情報の在処）

- ページ位置の状態は **サーバ側に永続化** される。クライアントが受け取るのは、その状態を参照する識別子のみ。
- 採用理由:
  1. クライアントから渡されるトークンの改ざんリスクを遮断できる
  2. トークン長が一定（UUID 文字列）になる
  3. 内部カーソル形式の変更がクライアント契約に影響しない
- 内部表現は「最後に返した投稿の `(create_time, post_id)`」を JSON シリアライズしたもの。
  これにより seek 方式のクエリを発行し、データ量が増えても性能が劣化しない。

## ページ整合性

- ページ走査中に新規投稿が追加されても、既に発行した `page_token` から見える結果集合は安定する
  （seek 方式のため、追加された投稿が以前のページを押し下げることはない）。
- 走査中に投稿が削除されても、削除済みは `valid=false` として結果に残るため、ページ境界がずれない。
- ただし、サーバ側に保持する `Pagination` レコードに有効期限は設けていない（明示的な失効は無い）。

## エラー

| 状況                                  | gRPC code         | REST status |
| ------------------------------------- | ----------------- | ----------- |
| `page_size` に **負数** を指定         | `INVALID_ARGUMENT` | 400         |
| `page_token` が UUID 文字列でない     | 実装上拒否される（内部エラー） | 5xx |
| `page_token` に対応する状態が存在しない | `NOT_FOUND`        | 404         |

## 利用例

### 初回リクエスト

```http
GET /api/v1/posts/list/10/   HTTP/1.1
```

`page_token` を空にしたいとき、URL パス末尾の空文字をどう表すかは
クライアントの REST 実装に依存する（gRPC では空文字を素直に指定できる）。

### 2 ページ目以降

```http
GET /api/v1/posts/list/10/3fa85f64-5717-4562-b3fc-2c963f66afa6 HTTP/1.1
```

`next_page_token` で受け取った値をそのままパスに含める。
